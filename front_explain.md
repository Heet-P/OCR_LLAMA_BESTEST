## ðŸ—ï¸ Architectural Overview & Core Components

 The application is structured as a **Single Page Application (SPA)** with a dynamic split-view layout.

### 1. `App.tsx` ( The Orchestrator)
The central nerve center of the frontend. It manages the global application lifecycle:
*   **State Machine**: Tracks the complex document lifecycle (`idle` -> `uploading` -> `processing` -> `completed` -> `error`).
*   **View Routing**: Manages the transition between the logical views (`home` vs `history`) without full page reloads.
*   **Event Handling**: Coordinates events between the Chat, Document Viewer, and Toolbar.

### 2. `UploadSection.tsx` (Seamless Ingestion)
The entry point for the user.
*   **Drag-and-Drop**: Intuitive file handling area.
*   **Real-time Progress**: Visual feedback bar that tracks upload percentage and backend processing phases (OCR -> Analysis -> Ready).
*   **Error Handling**: Graceful error boundary that informs the user of specific upload failures.

### 3. `ChatSection.tsx` (The AI Copilot)
This is where the magic happens. It's not just a chatbox; it's a context-aware form filler.
*   **Conversational Flow**: Renders the AI's questions based on the form's schema.
*   **Smart Validation**: Provides immediate feedback (via `ChatService`) if user input is invalid for a specific field.
*   **Auto-Scroll**: Automatically keeps the latest interaction in view for a fluid experience.

### 4. `DocumentViewer.tsx` (Visual Context)
A critical feature for user trust.
*   **Split-Screen Layout**: sits side-by-side with the chat, allowing users to verify information.
*   **Smart Highlighting**: (MVP Feature) Capable of visually indicating which section of the form is currently being discussed.
*   **Canvas/Image Rendering**: Renders high-fidelity PNG previews of the PDF pages generated by the backend.

### 5. `HistorySection.tsx` (Persistence)
*   **Dashboard View**: Displays a grid of previously processed forms.
*   **Status Badges**: Color-coded indicators (`Ready`, `Processing`, `Error`) for quick status checks.

## ðŸ”„ Key Workflows & Processes

### ðŸ”„ 1. The Real-Time Polling Mechanism
Since OCR and AI analysis are heavy compute tasks, we implemented an asynchronous status check pattern:
1.  **Upload**: File is sent to `POST /upload`. Backend returns a `form_id` immediately.
2.  **Poll**: Frontend enters a polling loop, calling `GET /forms/{id}/status` every 2 seconds.
3.  **Logs**: Real-time logs ("Identifying text...", "Analyzing structure...") are streamed to the UI to keep the user engaged.
4.  **Completion**: Once status is `ready`, the UI seamlessly transitions to the Chat/Document view.

### ðŸ”„ 2. Dynamic History Navigation
We implemented a custom navigation flow that allows users to access their past work without losing context. The `FloatingToolbar` provides quick access to reset the state or jump to history, keeping the UI clutter-free.
